Before you startFive facilitation principles

Hands-on from minute one. Concepts land through doing, not slides. Cut your talking; get them on a keyboard fast.

Teach the trust tool once. Introduce it in Session 1, then point back to it all afternoon. "This is the same grid — this morning it asked 'do I trust this output?'; now it asks 'where does a human stay in the loop?'"

Honesty builds trust. Be upfront about confidently-wrong output, the demo trap, and that killing a project is success. The credibility comes from not overselling.

Name the two scales. Make the morning→afternoon move explicit so the day feels coherent, not crammed.

End on the action plan. Vague plans don't survive Monday — make commitments specific, and have people swap them with a neighbour.

Activity · MorningThe two-pass demo

When: Session 2, ~11:20Duration: 15–20 minFormat: facilitator-led, live on projector

Purpose

The thesis, felt in the gut rather than heard as a quote. Participants watch a generic answer become uniquely good the moment human edge is added. This is the spine of the whole day — get it right.

Setup

  • Projector + an AI tool open (frontier model if possible — you want Pass 1 to be genuinely slick).
  • A real task with two versions ready: a naive prompt, and the same prompt enriched with something only a human holds (a specific customer, a contrarian bet, a taste call, domain knowledge the model lacks).
  • Follow-along optional — this works best as a watched demo, not a parallel exercise.

How to run it

  1. Pass 1 — generic. Run the naive prompt live. Let the output land. It will be slick, correct, polished. Point at it: "This is good. Every one of you would get this same good answer. So where's your edge?"
  2. The beat. Hold the silence. Let them feel that "good and identical" is not an advantage.
  3. Pass 2 — with edge. Layer in the human input. Run it again. Watch the output gain a soul — it now sounds like someone, not anyone.
  4. Make it interactive. Open the second pass to the room — let them shout the "edge" inputs (a customer, a constraint, a preference). The output visibly shifts with their input.

Debrief (3–4 min)

  • Where did Pass 2 stay generic, and where did your input make it irreducibly yours?
  • What's the part the tool can't manufacture — no matter how good the prompt?

Watch-fors

  • Pick a task where Pass 1 is already good. If the naive output is obviously bad, the "generic is the baseline" contrast collapses. You need it slick-but-identical.
  • Keep it to two passes. Don't let it drift into a prompting tutorial — the point is the contrast, not technique.
  • Companion-site pointer: the two-page voice method (Style Mirror) makes this edge persistent — mention it as the take-home, don't teach it here.

Worked example — the two prompts

Task: a difficult customer email (relatable; the contrast lands fast).

Pass 1 — generic:

"Write an email to a customer about their delayed order."

→ slick, polite, "we value your business," no specifics. Identical for everyone in the room. Hold it up: "Good. So where's your edge?"

Pass 2 — with edge:

"You're the CS lead. Write to Aisha, a 6-year VIP who's emailed twice this week, frustrated. Order #4471 is 10 days late — our freight partner lost the pallet; found yesterday, arriving Thursday. We've upgraded it to express and added a $40 credit. Tone: warm, no corporate filler, acknowledge she's annoyed. Under 120 words."

→ now it has a soul: specific, empathetic, references the real situation. The edge is everything highlighted — who the customer is, what actually happened, what you've already done, the tone this person needs. The model knew none of it; you supplied it. That's the part the tool can't manufacture.

Activity · MorningExercise 1 — the AI tool test drive

When: Session 1, ~10:10Duration: 20 minFormat: individual, then table debrief

Purpose

First hands-on contact. Each person runs a real task from their own work through an AI tool and classifies the output with the trust tool. Builds the habit of judging output before trusting it.

Setup

  • BYOD, an AI tool each (any). Network confirmed.
  • A fallback prompt ready for anyone who didn't bring a task (see Watch-fors).
  • Trust tool grid visible (slide or printed).

How to run it

  1. Set (2 min). "Pick a real, small task you genuinely need — an email, a summary, a first draft. Not a toy example."
  2. Do (8 min). Run it. Look at the output. Where on the grid does this task sit — average or precise? small or large stakes? Where would it be confidently wrong?
  3. Discuss at tables (optional, 5 min). Compare outputs across tools and tasks.
  4. Debrief (10 min). Draw out 3–4 people: one place the output was genuinely useful, one place it was confidently off. Pin each to a corner of the grid.

Debrief prompts

  • Did the output's confidence match its correctness? (It rarely does — that's the lesson.)
  • Which corner of the grid was your task in — and did you treat it that way?

Watch-fors

  • Have a fallback task. Some won't bring one. Suggested: "Draft a polite email chasing an overdue invoice" or "Summarise this article in 5 bullets." Keep it real but low-stakes.
  • Discourage the trivial ("tell me a joke") and the too-big ("write my strategy"). Neither teaches the grid.
  • Spot the confidently-wrong moment in the debrief — it's the highlight. Amplify it.

Worked example

Task to suggest: "Summarise these meeting notes into 5 action items" (paste your raw notes). Average is fine, small-medium stakes → lean in — but verify each action item actually matches: AI often invents an action nobody agreed to, or drops the one contentious point. That's your confidently-wrong moment.

Trust-tool call: Average + Small → lean in; the action items are the precise part, so sanity-check before you act.

Fallback task (if someone has nothing): "Draft a polite email chasing an invoice 14 days overdue, keeping the relationship intact."

Activity · MorningThe prompt library (RTCF)

When: Session 2, ~11:05Duration: 25 minFormat: individual, worksheet

Purpose

Build the habit of structured prompting and leave with a reusable set of prompts for real tasks. Land the key nuance: Role/Task/Format are scaffolding (most needed on weaker or self-hosted models); Context is the part that always matters — it's where your edge enters.

Setup

  • The Your prompt library worksheet (print or device). Three real tasks each.

How to run it

  1. Teach RTCF briefly with the before/after email example (unusable → sendable).
  2. Do (15 min). For three real tasks: write the naive version, then rebuild with RTCF. Emphasise filling in Context — the specifics only they hold.
  3. Run the two-pass mini-version on one prompt: thin context vs rich context. Watch the output improve.
  4. Debrief (8 min). Which component changed the result most? (Usually Context.)

The nuance to land

Be explicit: a frontier model follows plain, well-written instructions well — if you're good at briefing a human, you're most of the way there, so don't over-ritualise the labels. RTCF earns its keep most on smaller / self-hosted models. Context always matters — no model supplies it for you. The durable skill is "be clear about what you want," not "master a framework."

Watch-fors

  • Don't let it become prompt-engineering theatre. The goal is judgement about what to specify, not memorising an acronym.
  • Self-hosted participants (running local models) will see bigger gains from full RTCF — affirm that; it's where the scaffolding pays off most.

Worked example — naive → RTCF

Naive:

"Write a LinkedIn post about our new AI course."

RTCF:

"R: a confident but not hype-y B2B marketer. T: a LinkedIn post announcing The Human Edge, a 1-day Curtin exec-ed masterclass. C: audience = mid-career professionals sceptical of AI hype; the differentiator is judgement, not tools. F: hook line, 3 short paragraphs, end on a question, ~120 words."

The output jumps from generic to on-message. The component that did it: Context (the audience + the differentiator) — your edge. That's the beat to draw out in the debrief.

Activity · MorningExercise 2 — redesign your workflow

When: Session 2, ~11:50Duration: 25–30 minFormat: individual, worksheet

Purpose

Apply the trust tool to a real, repeated task the participant owns — and leave with a redesigned workflow they can trial next week. This is the morning's payoff: not "use AI more," but "use it deliberately, exactly where it belongs."

Setup

  • The Redesign your workflow worksheet. One real, repeated task each (something they own).

How to run it

  1. Name the task and list its steps (aim for 4–8).
  2. Judge each step with the trust tool: average or precise? small or large stakes? (Column by column on the sheet.)
  3. Redesign: let AI handle the average, low-stakes steps; protect the precise, high-stakes ones where judgement stays in charge.
  4. Pick one step to trial this week — and how they'll know if it worked.
  5. Debrief. Where did people over-trust? Where did they under-trust and protect too much?

Debrief prompts

  • Which step did you most want to hand to AI — and should you have?
  • Which step must stay yours, no matter how good the tool gets?

Watch-fors

  • One workflow, not the whole job. Scope-creep kills the exercise; keep them to a single repeated task.
  • Guard the high-stakes steps. Some will want to automate everything — use the grid to pull them back. The point is selective, deliberate use.
  • They leave with something concrete — make sure the "trial next week" commitment is specific.

Worked example — the weekly stakeholder report

Steps: (1) gather updates from the team · (2) draft the summary · (3) write the risks/decisions section · (4) format · (5) send.

Trust-tool split:

  • Gather updates → human (judgement on what's material; relationship).
  • Draft summary → AI (average, low stakes — lean in).
  • Risks/decisions section → human (precise, high stakes — judgement).
  • Format → AI (average, low stakes).
  • Send → human.

Redesign: AI drafts the summary from your raw bullet points and formats it; you own gathering (materiality) and the risks/decisions section. Trial this week: feed AI your raw bullets for the draft, then review.

Activity · AfternoonSprint 1 — scope it against reality

When: Session 3, ~1:30Duration: 45–60 minFormat: groups of 4–6, one initiative each

Purpose

Scope a funded RetailFlow initiative against reality by interviewing the people who hold it. Produce a scoped objective, data requirements, a trust-tool split, and a defined "good enough." This is differences 1 and 3 in action: "good enough" as a band, and data as discovery.

Setup

  • Each group takes one of the four initiative cards (no two groups alike).
  • RetailFlow portal open, code pilot2024. Sprint 1 worksheet each.
  • Interview Priya (Data) and Marcus (CIO).

How to run it

  1. Frame the boundary (2 min). The investment decision is made — the board funded it. This is delivery scoping (what ships first, what data it needs, what "good enough" means), not a business case. If a group drifts into ROI, pull them back.
  2. Interview (15–20 min). Marcus will push speed and ambition; Priya gives the honest, data-grounded timeline. Their scope is where the two reconcile.
  3. Capture (20 min): scoped objective (in/out of scope), data requirements from Priya, the trust-tool split of the work, and the single metric + threshold that defines "good enough."
  4. Debrief. Each group's scope + where they set "good enough" + how they reconciled Marcus vs Priya.

Debrief prompts

  • Where did you land on "good enough" — and could you defend it to a sponsor?
  • What surprised you in the data picture (Priya) vs the ambition (Marcus)?

Watch-fors

  • ROI drift. The most common trap. The card says it's funded; scoping ROI is a different conversation (Course 2).
  • Bots as consultants. Priya and Marcus are stakeholders, not your team — they give perspective and facts, not your scope. If a group asks the bot to "do the scope," redirect.
  • Scrutinise the bots. They're confidently wrong in places — that's intentional. It's the behaviour you're teaching them to lead.

Worked example — staff answer (Customer-service chatbot)

Scoped objective: deploy a chatbot to deflect routine queries (order status, returns, sizing) and cut email response time from 26h toward <4h. In: the top ~80% routine queries. Out (for now): complaints/escalations, phone, social DMs.

Data (from Priya): transcript data is the best of the four — but "good foundation" ≠ "knows your policy." Returns/sizing rules vary by product line and live in people's heads. The real work: RAG over current policy documents.

Trust-tool split: routine factual answers (order status) → lean in, with a human glance (it's customer-facing); policy answers (returns/sizing) → human in the loop — a wrong policy in writing is the live risk.

"Good enough": deflect ≥60% of routine queries; wrong-answer rate <2% on routine, but 0% tolerance on returns/policy (must route to a human if unsure).

Activity · AfternoonSprint 2 — stakeholders & human-in-the-loop

When: Session 4, ~3:05Duration: 45–55 minFormat: same groups, continuing their initiative

Purpose

Decide where a human must stay in the loop and design those checkpoints deliberately. This is the trust tool and difference 4 at project scale: verification is the product, not an afterthought.

Setup

  • Sprint 2 worksheet; groups continue their Sprint 1 initiative.
  • Interview Emma (MD), Tom (Customer Service), David (CFO).

How to run it

  1. Interview (15 min). Emma on strategy/board expectations; Tom on the frontline reality and customer impact; David on budget and the financial go/no-go.
  2. Build the stakeholder plan — who cares about what, and what each needs to hear.
  3. Design the human-in-the-loop checkpoints — where does verification sit, where must a person own the decision? Map them onto the workflow from Sprint 1.
  4. Debrief. Each group: where they put checkpoints, what's automated vs human-owned, and why.

Debrief prompts

  • Where did you default to "human everywhere" (losing speed) or "human nowhere" (losing trust)? How did the grid break the tie?
  • Which checkpoint, if you removed it, would the project quietly fail?

Watch-fors

  • The two failure modes — human-everywhere or human-nowhere — are both wrong. The trust tool exists to make this deliberate. Push groups to justify each checkpoint, not gut-feel it.
  • Tom is the reality check. Groups often under-weight the frontline; the human-in-the-loop design usually lives closest to Tom's world.
  • Checkpoints are part of the product, not an afterthought — make that explicit.

Worked example — staff answer (Customer-service chatbot)

Stakeholder plan:

  • Emma (MD/sponsor), High/High: needs the quarter promise; fears reputational damage → agree "good enough" gates + phased rollout, no big-bang.
  • David (CFO): needs ROI on $450K; fears runaway cost → tie spend to gates; show deflection savings.
  • Tom (CS/frontline), High interest: his team is automated first; fears job loss + looking bad → position as augmentation; staff checkpoints with CS.

HITL checkpoints:

  • Returns/policy answers → experienced CS reviewer checks before send (or bot refuses + routes). Evaluator: senior CS rep. Builds capability (mark this the deliberate build-checkpoint) + consumes.
  • High-confidence routine answers → QA-sample (~5%), human glance. Evaluator: CS lead. Mostly consumes.

The non-negotiable: returns-policy answers must be human-verified or refused — a wrong returns policy quoted to a customer in writing is the single biggest risk.

Activity · AfternoonSprint 3 — roadmap, risk & the go/no-go

When: Session 4, ~3:35Duration: 45–55 minFormat: same groups; ends in a decision

Purpose

Build the delivery roadmap with go/no-go gates and a risk register — then make the call: Scale, Pivot, or Kill. The capstone: every group defends a real decision under uncertainty.

Setup

  • Sprint 3 worksheet; groups continue their initiative.
  • Optional crisis element (staff resistance / data-quality / executive pressure) to stress-test the plan — use if time and energy allow.

How to run it

  1. Build the roadmap (15 min) — milestones, go/no-go gates, a risk register.
  2. (Optional) Crisis (10 min) — introduce one crisis; groups adapt the plan and checkpoint design. Great for energy in a long afternoon.
  3. Make the call (10 min): Scale, Pivot, or Kill — with a one-line rationale they can defend.
  4. Debrief. Each group announces their call + why. Normalise Kill as a legitimate, good outcome.

Debrief prompts

  • What was the single piece of evidence that drove your call?
  • If you had killed instead of scaled — what would you have saved? (And vice versa.)

Watch-fors

  • Nobody wants to Kill. It feels like failure. Reframe hard: killing a project that isn't working, early, is success; a portfolio that never kills can't learn. The cost of wrongly scaling is far higher than wrongly killing.
  • Make them defend the call, not just state it. The rationale is the learning.
  • Time-box ruthlessly. This is the last content block before the close — protect the 4:00 action-plan finish.

Worked example — staff answer (Customer-service chatbot)

Roadmap with gates:

  • Pilot (narrow): order-status + returns, shadow mode (bot drafts, human sends), 2 weeks. Gate: ≥70% of drafts usable with <3% wrong.
  • Evaluate: add sizing; measure deflection + wrong-rate. Gate: deflection ≥60%, wrong <2%, policy-wrong = 0.
  • Scale / Pivot / Kill → Production: full rollout with monitoring.

Risk register (top): confident-wrong returns policy (AI-specific: non-determinism + verification) — Med/High — mitigate: RAG + human checkpoint, bot refuses if unsure. · Demo-gap (AI-specific): the demo handles the easy 80% — mitigate: gate on the angry-customer edge cases.

Crisis (example): a customer screenshots the bot quoting a wrong returns policy. DDCD: Diagnose = technical + ethical; Decide = pause bot on returns, route to human, fix the RAG doc; Communicate = to the customer (apologise + honour), Tom's team, Emma; Document = root cause + fix.

Gate-1 call (model): PIVOT if the returns wrong-rate is too high at gate 1 — narrow scope to order-status only until the policy RAG matures. Rationale: the cost of a wrong policy at scale > the cost of narrowing scope. (Scale only if policy-wrong = 0 holds.)
